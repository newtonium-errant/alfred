## Summary

Janitor's deep sweep no longer drifts. The `last_deep + 24h` rolling
gate is replaced with `compute_next_fire(deep_sweep_schedule, last_deep)`,
scheduled to 02:30 America/Halifax daily. Light (structural) sweeps
stay on their 1h rolling cadence — this is exactly what we want.

Yesterday the deep sweep fired at 11:22 ADT because of a daemon
restart. From today on, the only way it fires outside 02:30 is if the
state file's `last_deep_sweep` is more than 24h stale — e.g. a
multi-day outage. That failure mode is acceptable.

## What changed

- `src/alfred/janitor/config.py`
  - Imported `ScheduleConfig` from `alfred.common.schedule`
  - Added `SweepConfig.deep_sweep_schedule` with defaults 02:30 /
    America/Halifax / daily (day_of_week=None)
  - Kept `deep_sweep_interval_hours` in place (unused but preserved
    for backward-compat with old config.yaml files that haven't been
    migrated — no crash if someone upgrades and forgets to update
    config)
  - Registered `deep_sweep_schedule` in `_DATACLASS_MAP` so the
    recursive `_build` picks up the nested dataclass
- `src/alfred/janitor/daemon.py`
  - Imported `compute_next_fire`
  - Removed the `hours_since_deep >= deep_interval_hours` check
  - On first boot, seed `last_deep = now` (instead of `datetime.min`)
    so the first scheduled fire is at the next 02:30, not immediately
  - New gate: `now >= compute_next_fire(deep_schedule, last_deep)`
  - Logs the next scheduled fire after each successful deep sweep
    (`daemon.deep_sweep_next` event)
  - Dropped unused `ZoneInfo` import
- `config.yaml` — added `deep_sweep_schedule` block under
  `janitor.sweep`, removed `deep_sweep_interval_hours`
- `config.yaml.example` — same, with an expanded comment pointing to
  the fallback behavior
- `tests/test_janitor_schedule.py` — 7 tests covering:
  - Default ScheduleConfig values
  - load_from_unified picks up the nested schedule dict
  - Missing schedule block falls back to defaults
  - Fresh-state seeding doesn't re-fire immediately
  - Clock crossing 02:30 fires (before/at/after 02:30)
  - Restart within the same scheduled window does not re-fire
  - Mixed UTC/Halifax timezone handling in the comparison

## Design decisions

- **Seed `last_deep` to `now` on first boot.** The rolling approach
  used `datetime.min`, which worked because `last + 24h` was always
  less than `now`. With `compute_next_fire` from `datetime.min`, the
  "next fire" is ~year 1 — `now >= that` is always true, so we'd
  deep-sweep on every boot. Seeding to `now` means the first fire is
  at the next scheduled window, which is the intended semantics.
- **Keep `deep_sweep_interval_hours` as a vestigial field.** If a
  user's config.yaml still has that key but no `deep_sweep_schedule`,
  the dataclass still loads (the value is ignored in the daemon
  loop). Avoids a hard break during upgrade.
- **Structural (light) sweeps stay rolling.** Only the LLM-heavy deep
  sweep was causing work-hour collisions; the 1h structural scan is
  cheap and benign.

## Alfred Learnings

- **Dataclass map registration matters.** The `_build` recursive
  dataclass constructor keys off `_DATACLASS_MAP`. Forgetting to add
  a new nested dataclass key there means the config loads as a plain
  dict and blows up when the daemon accesses `.time` on it. Caught
  this before tests — added `deep_sweep_schedule: ScheduleConfig` to
  the map in the same edit.
- **Pattern confirmed** — the ``compute_next_fire(schedule, last_deep)``
  check is cleaner than the rolling interval. No need to track
  "hours_since"; the helper encapsulates the policy.
