# Hotfix c3 2026-04-21

## Intent

Third of three hotfix commits. Two router bugs:

1. **Incomplete cue list.** Smoke test: "Check the output of pytest tests/transport/ -x" → classified `note` instead of `peer_route`. Current cues were too narrow ("run the tests", "look at the diff"); missed direct pytest invocations, npm commands, debugging phrases, git inspection, and direct-address patterns like "KAL-LE, X".
2. **No instance self-awareness.** The classifier prompt had no concept of "who am I?" — so KAL-LE's own classifier could happily emit `peer_route target=kal-le` when Andrew said "KAL-LE, run pytest" directly to KAL-LE's bot. Self-referential nonsense, and c1's dispatch guard only catches it at dispatch time (after the router's API call already cost a round-trip).

## Work Completed

### `src/alfred/telegram/router.py`

- **Expanded `peer_route` cue list** in the prompt (approx doubles the cue coverage):
  - Direct addressing: "KAL-LE, X" / "KAL-LE: X" / "K.A.L.L.E., X"
  - Test running: "run pytest", "run the tests", "check the tests", "check the output of pytest", "pytest tests/X", "npm test", "npm run lint"
  - Test debugging: "fix the broken test", "debug this test", "trace the failure", "why is X test failing"
  - Code work: "write a function", "refactor this", "add a test", "review the last three commits"
  - Git inspection: "git status", "git diff", "what's on this branch"
  - Aftermath-lab curation (unchanged from earlier)
- **Added instance self-awareness block** to the prompt:
  - Templated `{self_name}` / `{self_display_name}` placeholders
  - Instruction: "NEVER classify peer_route with target="{self_name}" — that would route to yourself"
  - Instruction: "If the user addresses your own instance by name (e.g., "{self_display_name}, ..."), strip the address and classify the remaining content normally"
- **Function signature change.** `classify_opening_cue(...)` now accepts `self_name: str = "salem"` and `self_display_name: str = "Alfred"` (defaults preserve legacy-caller behaviour). `_decision_from_parsed(parsed, recent, self_name="")` similarly accepts the local name for parse-time guarding.
- **Parse-time phantom self-target guard.** Even with the prompt instruction, the classifier can still emit `peer_route target=<self>`. The parse-time guard coerces to `note` and logs `talker.router.peer_route_self_target_coerced` at WARNING so we can track how often the classifier ignores the instruction.

### `src/alfred/telegram/bot.py`

- **`_open_routed_session` now threads instance identity into the router call.** Uses `_normalize_instance_name(config.instance.name or canonical or "salem")` for `self_name` (matches c1's `_dispatch_peer_route` convention). `self_display_name` uses the canonical form first so the user-facing example reads naturally ("K.A.L.L.E., ..." not "KAL-LE, ...").

### `tests/telegram/test_router_peer_route.py` (+23 tests)

- **`test_coding_cues_classify_peer_route`** — parametrized test over 18 new cue strings. Confirms each produces `peer_route` when the classifier cooperates. Also sanity-checks that a sample of the new cues (`run pytest`, `npm test`, `git status`) appear verbatim in the prompt passed to the SDK.
- **`test_direct_address_from_salem_routes_to_kalle`** — "KAL-LE, run pytest" from Salem classifies as `peer_route target=kal-le`.
- **`test_direct_address_from_kalle_self_coerces_to_note`** — same message on KAL-LE's own classifier (with `self_name='kal-le'`) degrades to `note` via the parse-time guard.
- **`test_self_address_on_salem_stripped_and_classified_normally`** — "S.A.L.E.M., tell me a joke" on Salem classifies as `note` (not peer_route), and the prompt carries the self-awareness instruction.
- **`test_self_name_parameter_renders_in_prompt`** — guards the `{self_name}` / `{self_display_name}` placeholders against accidental removal.
- **`test_default_self_name_preserves_legacy_behaviour`** — calling `classify_opening_cue` without the new params still works.

## Testing

Full suite: 952 → 975 passing (+23). Existing router tests (continuation validation, parse-failure fallback, API-error fallback) all unaffected by the signature change because they accept the defaults.

## Alfred Learnings

- **Three-layer defense-in-depth for the self-target problem.** c1 catches it at dispatch time (bot.py guard); c3 catches it at parse time (router.py parse-time guard); c3 also preempts it at prompt time (instruction in the prompt template). The parse-time guard gives us observability — if the WARNING fires regularly, the prompt instruction is being ignored and we need to revisit the prompt. If it never fires, the prompt is working as intended. Pattern worth keeping: instrument the fallback paths so we can see whether the preventive layer actually works.
- **Router cue lists rot fast.** Six cue strings landed in the initial Stage 3.5 ship; within a day, the live smoke test found at least three missing patterns (direct pytest invocations, npm commands, git inspection). Cue lists need a dogfooding pass per deployment session — we can't predict natural-language patterns without real usage. Tracking idea for later: surface the last N `note`-classified messages so we can audit for routing misses.
- **Prompt placeholders need tests.** `{self_name}` / `{self_display_name}` are easy to accidentally delete during a prompt rewrite. The `test_self_name_parameter_renders_in_prompt` guard asserts they land in the outgoing prompt — this is the second time we've added such a test (the first was for `{{instance_name}}` in the SKILL). Pattern confirmed: every placeholder in a prompt template gets its own presence-assertion test.
- **Default-preserving function signatures let us ship incrementally.** `classify_opening_cue` gained two params but kept every existing test passing because they default sensibly. If we'd required the caller to pass them, this would have been a larger churn touching every test fixture.
