## Summary

Brief now consumes `alfred.common.schedule.ScheduleConfig` instead of
defining its own. Zero behavior drift — the YAML shape is unchanged
(`brief.schedule.time` + `brief.schedule.timezone`), `load_from_unified`
still accepts the same dict, and the daemon's next-fire output matches
the old `_next_run_time` helper exactly. Proves the abstraction on the
one tool that was already clock-aligned, unlocking the janitor and
distiller migrations in c3-c5.

## What changed

- `src/alfred/brief/config.py`
  - Removed the local `ScheduleConfig` dataclass
  - Imported `ScheduleConfig` from `alfred.common.schedule`
  - `load_from_unified` builds the shared dataclass with the same
    defaults (`06:00` / `America/Halifax`) and `day_of_week=None`
    (brief is daily-only)
- `src/alfred/brief/daemon.py`
  - Removed the local `_next_run_time` helper
  - Imported `compute_next_fire` from `alfred.common.schedule`
  - Main daemon loop now calls `compute_next_fire(config.schedule, now)`
  - Dropped the unused `timedelta` import
- `tests/test_brief_config.py` (new, 4 tests)
  - Confirms `cfg.schedule` is the shared dataclass instance
  - Defaults still resolve correctly
  - Custom values survive load
  - Daemon next-fire output matches `compute_next_fire` directly
    (zero-drift contract)

## Zero-drift contract

`_next_run_time(time, tz)` and `compute_next_fire(schedule, now)` emit
the same target time for identical inputs. The old helper used
`replace()` then bumped to tomorrow if `target <= now`; the new helper
uses strict-after semantics (`target > now` fires today, else tomorrow).
The behaviours are identical for all inputs where `now` is not
*exactly* `time` — and if it is, both roll forward to the next day.

Brief's outbound-push wiring (commit `a99592d`) is untouched —
`_push_brief_to_telegram` was not in the refactor path.

## Alfred Learnings

- **Pattern validated** — the refactor was genuinely mechanical once
  the shared primitive was in place. ~30 lines removed, ~5 added,
  same semantics.
- **Test technique** — "daemon next-fire matches helper output"
  is the right shape for zero-drift migration tests. Don't re-test
  the helper's behavior; just assert the caller is wired to it.
