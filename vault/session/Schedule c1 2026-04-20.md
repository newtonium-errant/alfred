## Summary

First commit of the scheduling consolidation arc. Introduces
`src/alfred/common/schedule.py` with a `ScheduleConfig` dataclass
(daily + weekly shapes) and a pure `compute_next_fire(cfg, now)`
helper. No tool is migrated yet — this is the abstraction on its own
so subsequent commits can swap rolling-interval gates for clock-aligned
scheduling on brief, janitor deep sweep, distiller deep extraction, and
distiller consolidation.

## Why

Janitor deep sweep (`last_run + 24h`) drifts every daemon restart.
Yesterday's sweep landed at 11:22 ADT, right during Andrew's working
window. Heavy passes should be overnight. Brief already does this with
`schedule.time: "06:00"` + timezone — this factors out that pattern as
a reusable primitive.

## What changed

- New package: `src/alfred/common/` (empty `__init__.py` for now)
- New module: `src/alfred/common/schedule.py`
  - `@dataclass ScheduleConfig(time, timezone, day_of_week=None)`
  - `compute_next_fire(config, now) -> datetime` — requires tz-aware
    `now`, returns the next fire in the config's timezone
  - `parse_day_of_week(s) -> int` — monday-sunday, case insensitive
- New tests: `tests/test_schedule.py` (19 tests)
  - Daily fires-today and fires-tomorrow
  - UTC input is converted to target tz before comparison
  - Weekly gate from every starting weekday lands on the target
  - Halifax DST spring-forward and fall-back both resolve cleanly
  - Naive `now` input raises (pure-function contract)
  - Invalid time / day_of_week raise

## Design decisions

- Strict-after semantics: if `now == target`, we return tomorrow.
  Prevents a double-fire if the caller sleeps for zero seconds and
  loops back.
- No UTC conversion in the return value — we return the datetime in
  the config's tz. The caller can `.astimezone(timezone.utc)` if
  needed. Easier to eyeball during debugging.
- DST: construct target datetimes inside the zone via
  `datetime(..., tzinfo=tz)` so zoneinfo resolves ambiguous/skipped
  slots. The spring-forward skipped hour shifts forward; the
  fall-back ambiguous hour returns the first occurrence.
- No cron expressions. Two shapes (daily + weekly) cover every
  operational pass alfred runs. Add shapes when a concrete use case
  demands them.

## Alfred Learnings

- **Pattern validated** — the brief daemon's existing next-fire
  computation (`_next_run_time` in `src/alfred/brief/daemon.py`) is
  the correct shape. Generalising it was mechanical; the only
  interesting bit was the weekly case.
- **Gotcha** — `datetime.replace(tzinfo=...)` preserves the literal
  wall-clock fields but does NOT re-resolve DST. Construction via
  `datetime(year, month, day, hour, minute, tzinfo=tz)` is the
  DST-safe form. Noted inline in the module docstring.
- **Test approach** — picked 2026 DST transitions as concrete dates
  (spring forward 2026-03-08, fall back 2026-11-01) so tests don't
  silently stop exercising DST when the year rolls over.
