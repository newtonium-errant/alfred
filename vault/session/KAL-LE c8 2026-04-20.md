  - Prints BotFather checklist to stdout with the exact env var names.
  - `--force` flag to overwrite an existing config.
  - Rejects empty/invalid names.

## Deviations from plan

- **Path substitution is naive string replace** — if someone names their instance "kalle-2026-04-20", the "kalle" token gets replaced across the whole file. Acceptable for scaffolding; operator reviews + tunes the output per the step-4 instruction in the CLI's printed checklist. If this becomes a footgun, switch to Jinja templating.
- **KAL-LE template is the only template** — STAY-C will either reuse this (tuning identity + PHI flags) or get its own when it actually ships. Not forcing a template abstraction layer before the second instance exists.

## NOT done in this commit

- **No actual KAL-LE standup** — the template + CLI are in place; the user runs through the BotFather steps + token generation + `alfred instance new kalle` afterward. Per task instructions: "Do NOT actually stand up KAL-LE yet — the team lead + user will do that after all 11 commits land and BotFather is done."

## Alfred Learnings

- **Pattern validated** — scaffolding CLIs that print checklists to stdout rather than trying to automate everything. The BotFather step is inherently interactive (new-bot naming, token capture); `alfred instance new` prepares everything around it and lets the user fill in the human-only parts.
- **Gotcha** — `config.kalle.yaml.example` is found via two search paths (cwd + project source tree) so the CLI works from any working directory. The naive cwd-only lookup would have failed when Andrew runs from `~/.alfred/kalle/` after standing it up.
- **Missing knowledge** — the `/home/andrew/.alfred/` root needs to exist before the CLI can create subdirs. First-run UX is "create ~/.alfred" as a one-time prerequisite. Could be auto-created by this CLI but chose not to because it implies a permissions assumption we shouldn't make. Flag for quickstart if anyone else tries this path.
