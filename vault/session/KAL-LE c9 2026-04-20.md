# KAL-LE c9 — BIT per-peer probes + `alfred check --peer`

## What shipped

### `transport/health.py` — new probes

- `_check_peer_reachable(raw, peer_name, peer_entry)` → `peer-reachable:<name>` check. GETs the peer's `/health` endpoint (unauth'd bootstrap probe). OK on 200 JSON; **WARN** on connection refused (peer daemon down is not a FAIL — the other side could be legitimately offline); FAIL on non-JSON or non-200 from a reachable port.
- `_check_peer_handshake(raw, peer_name, peer_entry)` → `peer-handshake:<name>`. POSTs `/peer/handshake` with the caller's bearer + instance identity. OK on 200 + matching `protocol_version`; **WARN** on version skew; FAIL on 401 auth rejection or missing token.
- `_check_peer_queue_depth(raw, peer_name)` → `peer-queue-depth:<name>`. Inspects the local transport state file's `pending_queue`, counts entries whose `peer` field matches. 100-entry WARN threshold.
- `_infer_self_name(raw)` — resolves `telegram.instance.name` to a peer-shaped identifier. "Alfred" → "salem"; "S.A.L.E.M." → "salem"; "KAL-LE" → "kal-le".
- `_run_peer_probes(raw, filter_peer=None)` — iterates `transport.peers` and runs all three probes per peer (or narrows to one).
- **`health_check(raw, mode, filter_peer=None)`** extended:
  - When `filter_peer` is set, the **local** transport probes skip entirely. Only the named peer's probes run.
  - Unknown peer → SKIP with detail "no peer named ...".

### `cli.py` — `alfred check --peer <name>`

- New `--peer` argument on the `check` subparser.
- When set, `cmd_check` **bypasses** the aggregator and calls `transport.health.health_check(raw, mode, filter_peer=peer)` directly, wrapping the result in a single-tool `HealthReport`. Streaming/JSON rendering identical to the normal path.

### `transport/cli.py` — `alfred transport tail [--peer <name>] [--limit N]`

- New `tail` subcommand. Reads `canonical_audit.jsonl` via the `canonical_audit.read_audit` helper. Filters entries where `entry["peer"] == peer` when `--peer` is set. Renders each entry as a one-liner with timestamp, peer, type, name, granted + denied lists, and correlation id.
- Designed so operators can grep a specific peer's access pattern: `alfred transport tail --peer kal-le --limit 100`.

## Tests

- 19 new tests in `test_peer_health_probes.py`: reachable OK / WARN / FAIL; handshake OK / version-skew WARN / missing-token FAIL / unreachable WARN; queue-depth state-missing / matching-peer-counts / threshold-WARN; `_infer_self_name` for three instance names; `_run_peer_probes` filter behaviour; `health_check` with filter + unknown-peer SKIP; `cmd_tail` peer filter + no-filter.
- Suite: 914 → 933.

## Deviations from plan

- **`peer-auth` probe** — plan listed it as a separate probe but the auth check is already the 401 branch of `_check_peer_handshake`. Splitting it into a second probe would issue a second `/peer/handshake` call; the combined check covers both. Flagging so plan → code expectations align.
- **`--peer` flag** wires through to a fresh transport-only health-check call (bypassing the full aggregator) rather than adding a `tools=["transport"]` filter. Aggregator filter is whole-tool, not per-peer, so bypass was the cleaner fit.

## Alfred Learnings

- **Pattern validated** — "WARN on connection refused" for peer probes. A peer being down is a normal state, not a failure. FAIL is reserved for "the peer is up but misbehaving."
- **Pattern validated** — `_infer_self_name` → `salem`/`kal-le` is derived from config, not env. The instance's identity is knowable from `telegram.instance.name` alone, no separate config required.
- **Missing knowledge** — the `pending_queue` entries only carry a `peer` field because Stage 3.5's outbound-write path (server.py `_handle_send`) writes `request.get("transport_peer")` onto each queued entry. Without that write, `_check_peer_queue_depth` would always return 0. Confirmed present on c3 shipped code; if anyone refactors the outbound handler, the probe breaks silently. Flagged.
