# KAL-LE c1 — Multi-instance config plumbing

First commit of the 11-commit KAL-LE MVP arc. Makes `alfred up` multi-instance-capable without standing up KAL-LE itself.

## What shipped

- **`daemon.pid_path`** top-level config field. When set, overrides the legacy `<logging.dir>/alfred.pid`. Prevents Salem and KAL-LE (both on the same WSL2 host) from fighting over one pid file.
- **`telegram.instance.skill_bundle`** — default `"vault-talker"`. The talker daemon now reads this and loads `skills/<skill_bundle>/SKILL.md` at startup. Salem unaffected; KAL-LE will override to `"vault-kalle"` in c5.
- **`telegram.instance.tool_set`** — default `"talker"`. Round-tripped through config now; wired into the tool schema in c5/c6.
- `_resolve_pid_path()` helper in `cli.py`. Used by `cmd_up`, `cmd_down`, `cmd_status`, `cmd_tui` so they all agree on the path.
- `config.yaml.example` opens with a `# Multi-instance (Stage 3.5) setup` block documenting the port convention (8891..8895) and pointing at `config.kalle.yaml.example` (arrives in c8).

## Tests

- 8 new tests in `tests/test_multi_instance_config.py` covering pid_path defaulting, skill_bundle override, tool_set round-trip, and `_load_system_prompt` resolving to the right bundle directory.
- Suite: 734 → 742, all green.

## Alfred Learnings

- **Pattern validated** — the existing `_substitute_env` + `_build` config pattern absorbed three new fields without churn. Adding per-instance config stays cheap.
- **Gotcha** — `cmd_tui` had its own ad-hoc `log_dir / "alfred.pid"` construction. Centralising into `_resolve_pid_path` is the only way to keep every CLI entry point honest about multi-instance paths; next time anyone adds a pid-path reader, they need to grep for `alfred.pid` and make sure it flows through the helper.
- **Missing knowledge** — the orchestrator and daemon spawn pattern was clear; what wasn't documented until now is that `logging.dir` is the single source of truth for every per-process artefact (pid file, logs, state files). `daemon.pid_path` is the first field to explicitly decouple; others (state paths) could follow for tighter per-instance isolation.
