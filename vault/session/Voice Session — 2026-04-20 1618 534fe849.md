## Typographic Convention

Two punctuation styles have emerged, and they carry meaning:

- Hyphens (STAY-C, KAL-LE) → specialist instance, narrow domain, does one thing well
- Periods (S.A.L.E.M., V.E.R.A.) → broad-scope / orchestrator-tier instance, coordinates across many domains or handles a whole business unit

This wasn't designed — it emerged from the first four names and is worth preserving for future instances. When naming a new instance, the punctuation choice should signal its scope at a glance.

## Input Normalization for Routing

The stylized canonical names won't survive phone typing or audio transcription intact. The router must accept the realistic variants users will actually produce in normal use.

### Canonical → Accepted Aliases (case-insensitive)

| Canonical | Accept Aliases |
|-----------|----------------|
| STAY-C | Stacy, Stacey |
| KAL-LE | Kali, Callie, Kallie |
| S.A.L.E.M. | Salem |
| V.E.R.A. | Vera, Verra, Vira |

### Why this set

Phone autocorrect pushes input toward dictionary words and common name spellings. Audio transcription (Whisper, etc.) does the same — it hears syllables and writes the most familiar name that matches.

- "STAY-C" spoken aloud sounds like "Stacy." Autocorrect on phones will fight any attempt to type the hyphenated form.
- "KAL-LE" (KAL-lay) doesn't match any common name, so transcribers default to nearby names: Kali, Callie, Kallie. Phone typing gravitates to Kali.
- "Salem" is already a recognized word/name. Both autocorrect and transcription land on it cleanly with no variants worth handling.
- "Vera" is a common name; Whisper occasionally guesses "Vira" or doubles the r.

### Implementation guidance

- Normalization happens on the input side — route requests to the correct instance regardless of which alias the user used.
- Match should be case-insensitive.
- Instances should refer to themselves by canonical form (STAY-C, KAL-LE, S.A.L.E.M., V.E.R.A.) in logs and outbound responses, but accept any alias on inbound.
- Don't bother handling rarer variants until they actually appear in user input. This list is the minimum to feel natural in normal phone and voice use.
- When a new instance is confirmed, add its canonical form and predicted phone/voice aliases to the table above as part of the same commit.

## Naming Heuristics for Future Instances

1. The pun should be load-bearing — the name should tell you what the instance does
2. Pop culture references preferred; music references especially welcome
3. Multiple layers of meaning are good (public reference + personal resonance + functional description + etymology/secondary meaning of the bare word)
4. Don't force coherence across the roster — Foundation, Sabrina, Prozzak, and Vera Lynn don't need to share a universe
5. The "Temu Tony Stark" sensibility applies — scrappy, self-aware, the names can acknowledge the duct-tape-and-ambition nature of the system
6. Consider the phone/voice alias question at the time of naming, not after — if a stylized canonical form will produce predictable transcription variants, document them when the name is locked in
7. Choose the punctuation style (hyphen vs. periods) deliberately — it signals scope tier (specialist vs. broad-scope/orchestrator) and should match the instance's actual role
