# KAL-LE c4 — Client peer dispatch + correlation IDs

## What shipped

- **`client.py` additions**: `peer_send(peer_name, kind, payload, ...)`, `peer_query(peer_name, record_type, name, ...)`, `peer_handshake(peer_name, ...)`. Resolves peer base_url + token from `config.peers[peer]`. Same retry shape (0.5s → 2s on 5xx/connection, never on 4xx) as the outbound path via a new `_peer_request` helper.
- **`peers.py` (new)** — in-memory correlation-id response inbox. `register_response(cid, reply)` delivers to a waiter (event-set) or parks in the orphan buffer (OrderedDict, max 128, 5-min TTL). `await_response(cid, timeout=60)` blocks on the event or picks up an existing orphan. `inbox_stats()` exposes waiter + orphan counts for the CLI.
- **Correlation IDs**: 16-char uuid hex generated client-side when absent, sent as `X-Correlation-Id` header + body field. Round-trips through handler response so retries are deduplicable.
- **`_resolve_peer()`** — clean error on unknown peer / empty token so a misconfigured peers dict surfaces loud, not as a 401 loop.
- **Audit path empty-string disable** — `append_audit("")` is a no-op. Added so `test_peer_handlers` fixtures don't create stray `./canonical_audit.jsonl` files in the repo root. Prod configs always set this; tests that assert on audit override it.

## Tests

- 14 new tests in `test_peer_client.py`: peer resolution (known/unknown/empty token), response inbox (deliver/timeout/orphan-pickup/TTL prune/stats), peer_send end-to-end against a stub server, peer_handshake, peer_query with fields, 5xx retry, 4xx no-retry.
- Suite: 776 → 790, **+1 pre-existing flaky** on `test_failure_log_has_subprocess_contract_fields` when run with `tests/health` tests (structlog renderer state leak between test modules). Reproducible on HEAD commit before my changes — flagged, not mine to fix in this arc.

## Deviations from plan

- Plan said "fold into client.py or new peers.py" — I put HTTP dispatch in `client.py` (co-located with `_request`/retry shape) and inbox in a new `peers.py`. Mixing them would have conflated "fetch from peer" with "reply-callback plumbing."

## Alfred Learnings

- **Gotcha** — `test_peer_handlers.py`'s fixture default wrote `./canonical_audit.jsonl` to the repo root because `CanonicalConfig.audit_log_path` defaulted to `"./canonical_audit.jsonl"`. Changed `append_audit` to treat empty path as no-op; tests that assert on audit now explicitly pass a tmp path. Pattern: any file-writing function called from test fixtures needs a "disabled" sentinel.
- **Pattern validated** — matching the outbound retry shape byte-for-byte (`_peer_request` mirrors `_request`) means one bug class rather than two. Net duplicated lines ~60; cognitive cost of diverging retry semantics would be much higher.
- **Flaky test flagged** — `test_failure_log_has_subprocess_contract_fields` asserts on `capsys.readouterr()` picking up structlog output. `tests/health` tests reconfigure structlog to a non-stdout renderer, so this test fails when it runs after them. Pre-existing on HEAD — will flag for code-reviewer or backlog.
