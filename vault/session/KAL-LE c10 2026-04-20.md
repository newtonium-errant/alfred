## What shipped

### `tests/integration/test_dual_instance.py` — 4 tests

1. **`test_cross_instance_handshake`** — Salem + KAL-LE built on ephemeral ports. Salem's client hits KAL-LE's `/peer/handshake` and gets the right identity + protocol version + correlation-id echo.
2. **`test_salem_canonical_query_filters_fields`** — Salem hosts a real canonical person record with full frontmatter (name, email, phone, timezone, preferences). KAL-LE queries over `/peer/query` requesting `["name", "email"]`. Response has only those fields; phone+timezone+body all absent. Audit log entry created with correct granted/denied.
3. **`test_salem_forwards_to_kalle_and_receives_reply`** — the full loop:
   - Salem sends `/peer/send` to KAL-LE with correlation_id
   - KAL-LE's inbox callable receives it, simulates a bash_exec result
   - KAL-LE POSTs `/peer/send` back to Salem with `kind="query_result"` + same correlation_id
   - Salem's inbox callable calls `register_response(cid, reply)`
   - A `await_response(cid)` coroutine wakes up with the reply — the exact shape Salem's router uses
4. **`test_fallback_when_peer_unreachable`** — `peer_send` to a dead URL raises `TransportServerDown`. Salem's router code catches this and falls through to its own LLM turn (covered in c7 tests).

## Peer-pair token convention made explicit

This commit's config shapes clarify the peer-auth model from c8's template:
- `ALFRED_KALLE_PEER_TOKEN` is what **Salem** sends when reaching KAL-LE. KAL-LE's `auth.tokens.salem.token` matches it.
- `ALFRED_SALEM_PEER_TOKEN` is what **KAL-LE** sends when reaching Salem. Salem's `auth.tokens.kal-le.token` matches it.
- Each instance's `peers[<name>].token` is the OUTBOUND token — what this side sends to reach that peer. Each instance's `auth.tokens[<name>].token` is the INBOUND token — what we expect the peer to send to reach us.

## Alfred Learnings

- **Pattern validated** — ephemeral ports (`aiohttp_client` auto-assigns) keep integration tests collision-free in CI. Never hard-code 8891/8892 in a test.
- **Gotcha** — in aiohttp's auth middleware, the token→peer lookup iterates `config.auth.tokens` dict order. If two entries share the same token value, first match wins. Peer-pair tokens MUST be distinct (which they'll naturally be in prod — each is 64-char random hex). Test configs that reuse tokens across entries were debugging noise until I separated them.
- **Gotcha** — the correlation-id inbox (`peers.py`) is module-level global state. Between tests, clear `_INBOX` + `_ORPHANS`; otherwise a pending wait from a previous test bleeds into the next. Test 3 clears them explicitly; refactoring to an object-scoped inbox would be a bigger change than this arc warrants.
- **Pattern validated** — peer-forward round-trip via correlation-id. Salem sends with a cid, KAL-LE echoes it back, Salem's inbox writes into the waiting event. No sticky session state between instances — each side is stateless w.r.t. the peer protocol, only the correlation-id connects a request to its reply. Matches the Stage 3.5 D8 "message-level routing" decision exactly.
