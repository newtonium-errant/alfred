# KAL-LE c7 — Salem router extension for peer_route

## What shipped

### router.py
- `RouterDecision` grew two fields: `target: str | None` (canonical peer name like `"kal-le"`) and `peer_route_hint: str` (optional one-liner for logs). Both default-safe — pre-Stage-3.5 callers still work.
- `_ROUTER_PROMPT` extended with a `peer_route` enum option + examples ("why is the transport scheduler firing twice", "refactor the janitor fix command", "run the tests", "review the last three commits", "promote this pattern to canonical"). Output schema widened to include `target` + `peer_route_hint`.
- `_decision_from_parsed` validates peer_route specifically: target must be in `_VALID_PEER_TARGETS = {"kal-le", "stay-c"}`, lowercase-normalized. Missing/phantom targets **degrade to `note`** rather than forwarding to nowhere.

### session_types.py
- Added `peer_route` entry to `_DEFAULTS_TABLE`: Sonnet model, `supports_continuation=True` (follow-up turns on the same coding session keep forwarding), `pushback_level=0` (Salem is a router for this type — pushback is KAL-LE's job).

### bot.py
- New `_dispatch_peer_route()` helper (+140 lines):
  1. Sends `"→ KAL-LE"` acknowledgment before dispatch.
  2. Loads TransportConfig from `bot_data["raw_config"]` (passed through build_app from daemon).
  3. Generates 16-char correlation ID, calls `peer_send(target, kind="message", payload={user_id, text, originating_session, chat_id}, ..., correlation_id=cid)`.
  4. Starts a 20s mid-wait ping task that DMs `"KAL-LE still working…"` if the peer is slow.
  5. Awaits the peer's callback via `await_response(cid, timeout=45s)`.
  6. Relays the reply as `"[KAL-LE] <text>"`.
- Fallback on `TransportServerDown`: apologetic message + return `False` so the outer handler falls through to Salem's normal LLM turn.
- `_open_routed_session` now stashes `_peer_route_target` on the active dict when the router picks peer_route.
- `handle_message` reads `_peer_route_target` **before** the LLM turn — if present, dispatches to peer. Auto-forward continues across turns until `/end` clears the active dict.
- `build_app` now accepts `raw_config: dict | None` and stashes it on `bot_data["raw_config"]` for `_dispatch_peer_route` to pull at forward time.

### daemon.py
- Passes `raw=raw` into `bot.build_app(..., raw_config=raw)` so the dispatcher can build a fresh TransportConfig at forward time.

### Constants
- `_PEER_MID_WAIT_PING_SECONDS = 20.0` (plan: 20s).
- `_PEER_MAX_WAIT_SECONDS = 45.0` (plan: 45s).

## Deviations from plan

- **Auto-forward clear on `/end`**: implemented by piggybacking on `state.pop_active(chat_id)` inside `close_session` — it drops the whole active dict, so `_peer_route_target` goes with it. No explicit clearing code needed.
- **Fallback path on TransportServerDown**: plan said "reply 'KAL-LE is offline' then fall through to Salem's own handling." Implemented as a user-visible message + `return False` from `_dispatch_peer_route` → the `handle_message` outer block clears `_peer_route_target` from the active dict + falls into Salem's normal LLM turn on the SAME message. So Andrew gets both the "offline" notice AND Salem's best-effort answer, rather than just the notice.

## Alfred Learnings

- **Pattern validated** — session-type routing lives in one classifier call. Adding peer_route was a prompt update + a target-validation gate, not a new code path. Adding STAY-C later is a single `_VALID_PEER_TARGETS` update + a prompt example.
- **Gotcha** — PTB's `bot_data` is the only way to pass per-process config into async handlers. Stashing `raw_config` at `build_app` time avoids re-loading `config.yaml` from disk on every turn (which would race with any hot-reload logic). New Stage 3.5 features that need config should use this same pattern rather than opening files in handlers.
- **Missing knowledge** — the correlation-id inbox from c4 is what makes the peer-route wait work. When KAL-LE receives the forwarded message, processes it, and POSTs `/peer/send` back to Salem with `kind="query_result"` + the same correlation_id, Salem's own peer-inbox callable (which still needs wiring in c11's dogfood) will call `register_response(cid, reply)`, unblocking the `await_response` Salem is sitting on. The whole loop is wired end-to-end in c10's smoke test.
