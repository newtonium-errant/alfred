# Voice Design Doc User Profile View — 2026-04-15

## Intent

Small followup to the earlier voice design doc revisions. The original migration plan for the existing untracked `vault/user-profile.md` said "migrate into the calibration section of `person/Andrew Newton.md` and delete the standalone file." The user clarified they want both paths: MVP collapses to single source of truth, long term re-introduces the top-level file as a convenience view without duplication.

## What Changed

### Two-stage migration plan for `vault/user-profile.md`

**MVP (with Stage 2a):** migrate the content of the existing untracked `vault/user-profile.md` into the `<!-- ALFRED:CALIBRATION -->` section of `person/Andrew Newton.md` and delete the standalone file. Single source of truth during the validation period. Every read and write goes through the person record.

**Long term (re-introduce after validation):** bring `vault/user-profile.md` back as a top-level convenience file that transcludes the calibration sections from `talker.primary_users` via Obsidian's native embed syntax:

```markdown
# User Profile — Aggregated View

## Andrew Newton

![[person/Andrew Newton#Alfred's Calibration]]
```

Zero duplication — transclusions are live views, not copies. No background regeneration job, no race condition between the canonical record and the aggregate file. The top-level file is discoverable ("open user-profile.md to see Alfred's mental model") but remains a live view of the canonical data.

**When to actually do the long-term re-introduction**: only if the user finds themselves navigating to `person/{user}.md` frequently to check Alfred's model. If they don't miss the top-level file, the MVP single-source-of-truth state is good enough to operate indefinitely. The doc is explicit about this — don't re-introduce it reflexively.

## Alfred Learnings

### Patterns Validated

- **Obsidian's native features beat custom infrastructure where possible.** My first instinct for "top-level file that shows Alfred's mental model" was an auto-generated file regenerated by a background task. That would have introduced a new regeneration job, a sync race window, and duplicate content to keep in sync. Obsidian's transclusion syntax makes all of that disappear — the aggregate file becomes a live view with no infrastructure cost. Rule for elsewhere: when designing a feature that touches the vault, check whether an Obsidian-native feature (transclusion, base views, dataview queries, frontmatter properties, canvas, links) already solves it before reaching for code.
- **"Do it when you need it" is the right default for convenience features.** The original plan was to migrate and delete. The user's answer preserves that as the MVP, then adds a conditional long-term step. The condition is "only if you find yourself navigating to the canonical record often." That's the right posture for any convenience feature — build the canonical thing, ship it, see what friction you actually feel, and only then add the convenience layer. Avoids building convenience for friction that doesn't exist.
