# Hotfix c1 2026-04-21

## Intent

Live KAL-LE smoke test surfaced three issues. This is the first of three hotfix commits: stop `_dispatch_peer_route` from firing the `→ KAL-LE` ack when the router classified `peer_route target=<self>`. Previously the ack fired first and a confusing "Couldn't reach KAL-LE" error followed — user saw handoff that never happened.

## Work Completed

### `src/alfred/telegram/bot.py`

- **New module-level helper** `_normalize_instance_name(s)` — collapses an instance name into the canonical peer-key form used by `transport.peers` (lowercase, dots stripped, spaces → dashes). Also applies the legacy `alfred` → `salem` mapping so a default-configured install still matches the `salem` peer key.
- **Self-target guard in `_dispatch_peer_route`** — computes `self_name` from `config.instance.name` (matches `transport.health._infer_self_name` convention — `name` is the peer-key source, not `canonical`) and compares the normalized `target`. When they match, logs `talker.bot.peer_route_self_target` at INFO and returns `False` WITHOUT sending the `→ KAL-LE` ack. Caller falls through to normal Salem handling, same as the `TransportServerDown` branch.
- **Removed duplicated normalization** — the old inline `self_name` derivation (~line 1100) is gone; the guard's computed `self_name` is reused verbatim for the `peer_send` call so the wire protocol and the self-check agree on the same canonical form.

### `tests/telegram/test_router_peer_route.py` (new file, 9 tests)

Covers the normalization helper (lowercase, dot-stripping, space-to-dash, legacy alfred→salem, empty-input tolerance) and the self-target guard (default target matches, uppercase-dotted target normalizes, legacy alfred default maps to salem self, cross-instance target still dispatches past the guard).

## Key Decision

**Use `name` not `canonical` for self-identity.** The first draft used `config.instance.canonical or config.instance.name` but that breaks for KAL-LE: `K.A.L.L.E.` normalizes to `kalle` (no dashes), which doesn't match the peer key `kal-le`. Switched to `name`-first to mirror `transport.health._infer_self_name`. `KAL-LE` (name) → `kal-le` (peer key) matches. Documented the reasoning inline so this doesn't regress later.

## Alfred Learnings

- **Two conventions were diverging for "self-identity":** `transport.health._infer_self_name` used `instance.name`, but the peer-route dispatcher in `bot.py` used `canonical or name`. For KAL-LE these produce different peer keys (`kal-le` vs `kalle`). Standardise on `name`-first across the talker so peer keys line up. If we ever need to revisit, canonical forms with dots will always require dashes-per-word normalization if they're to match peer keys.
- **Returning `False` to fall through to Salem's local handling is the right pattern for the self-target case** — it matches the existing `TransportServerDown` branch, so the surrounding code already handles the fall-through correctly (clears `_peer_route_target`, lets Salem answer). No new fall-through plumbing needed.
- **Pre-existing flaky test tracking:** `test_failure_log_has_subprocess_contract_fields` in `tests/test_transport_client.py` fails under certain orderings due to structlog handler state. Deselect it per session; do not "fix" by monkey-patching structlog globally.
