## Summary

Final commit of the scheduling consolidation arc. Distiller's weekly
consolidation pass is now clock-aligned to Sundays at 04:00
America/Halifax via `compute_next_fire` with `day_of_week="sunday"` —
the first use of the weekly gate introduced in c1. The pre-c5 rolling
168h interval is replaced; consolidation no longer fires on daemon
boot but waits for the next Sunday 04:00 window.

All daily LLM-heavy passes are now schedule-aligned. The overnight
choreography looks like this in local Halifax time:

```
02:30 — janitor deep sweep        (daily, LLM fix pipeline)
03:30 — distiller deep extraction (daily, LLM extraction per-source)
04:00 — distiller consolidation   (Sundays only, LLM meta-analysis)
05:55 — BIT preflight             (existing, bit.schedule.lead_minutes)
06:00 — brief                     (existing, brief.schedule.time)
```

Andrew wanted this because yesterday's deep sweep landed at 11:22 ADT
right when he was active. Heavy passes now land while he's asleep.
Light passes (1h janitor structural, 1h distiller scan) stay rolling
— they're cheap and non-disruptive.

## Contracts the next builder should know

- **`alfred.common.schedule`** is the canonical scheduling module.
  Any future clock-aligned scheduling need (mail accounts polling at
  fixed times, surveyor nightly cluster refresh if that ever becomes
  a thing, etc.) should import from here.
- **Vestigial fields** (`deep_sweep_interval_hours`,
  `deep_interval_hours`, `consolidation_interval_hours`) are
  preserved for backward-compat but are no longer consulted by the
  daemons. They can be removed in a future cleanup pass once all
  known config.yaml deployments have been migrated.
- **First-boot seeding** (`last_*= now`) is the required pattern. The
  alternative (`datetime.min`) makes every clock-aligned gate fire on
  every boot.

## Alfred Learnings

- **Overnight choreography.** Heavy daily passes should run between
  ~02:00-06:00 local time, staggered ~30-60min apart so rate limits
  don't hit (02:30 → 03:30 → 04:00 → 05:55 → 06:00 is current). This
  is the operational window — document it on any future daemon that
  does LLM-heavy work.
- **Weekly gate was simpler than expected.** The abstract
  ScheduleConfig.day_of_week field did the heavy lifting; the
  distiller daemon just added `consolidation_schedule =
  config.extraction.consolidation_schedule` and `next_fire =
  compute_next_fire(...)`. Weekly-ness is entirely in
  `compute_next_fire`.
- **Seed on first boot.** Migrated three daemons this session
  (janitor, distiller deep, distiller consolidation). All three
  needed the same `last_* = now_utc_init` seed on fresh state. If
  we add a fourth clock-aligned schedule, this pattern is mandatory.
- **Pattern confirmed: light-pass rolling + heavy-pass aligned.**
  Janitor's 1h structural scan and distiller's 1h light scan still
  run on `asyncio.sleep(interval_seconds)`. Only the deep/weekly
  passes are clock-aligned. This is the right split — light work is
  self-throttled and harmless; heavy work needs a wall-clock gate.
- **`_DATACLASS_MAP` registration gotcha (flagged twice in this
  arc).** Every new nested dataclass needs an entry in the tool's
  `_DATACLASS_MAP` or the recursive config builder loads it as a
  plain dict and the daemon AttributeErrors on `.time`. Caught
  pre-test both times. Worth documenting in builder.md: "adding a
  nested dataclass to a config: update _DATACLASS_MAP in the same
  edit." Filing this as a potential agent-instructions update.
