# Adaptive Turn-End (Endpointing) for Salem PWA Voice — Decision-Grade Scope

Grounding confirmed against source this session: the single turn-fire seam is `voice_stt.py:474` (`await self._on_utterance(text)`) inside the `EVENT_UTTERANCE_END` branch (`voice_stt.py:453-493`); the PCM snapshot at `voice_stt.py:459` is unconditional and sits above the min-chars gate; `endpointing` is baked once into the wss URL at `stt_deepgram.py:79`; `parse_deepgram_message` already splits `is_final` from `speech_final` at `stt_deepgram.py:103-104`; and the provider's `events()` async generator (`stt_deepgram.py:419-424`) is a bare `while True: ev = await self._events.get()` with no `CancelledError` handling — so the concurrency hazard the facets flag is real. All three facet inputs are mutually consistent on the mechanism; they diverge only on a few tuning defaults and on how aggressively to build the learning layer. Facet C (feasibility/MVP) returned null, so I reconcile its intended tension — "a smart static heuristic may solve 90% cheaply" vs. "the full per-user-learned system" — directly from the RECON and Facet B §5 material, both of which carry that thesis explicitly.

## 1. Recommendation and order

**Ship the content-aware HOLD/COMMIT gate with snappy-safe static defaults as Increment 1, with features-only per-user telemetry flowing into a unified voice-calibration corpus from day one. Defer the per-user LEARNING (Increment 2) behind a pre-registered go/no-go gated on that telemetry.** This is the same collect-first discipline already ratified for STT shadow-capture and V3.1 barge (`docs/v3_1_barge_calibration_scope.md`).

The key reconciliation of the Facet C tension: **the "cheap smart static heuristic" and "the mechanism" are the same artifact.** Increment 1 with static signal lexicons and a fixed base/cap hold *is* the smart static heuristic — it is what fixes the observed friction now, at zero added latency on crisp speech. The per-user learner is not required to resolve the friction; it is a later refinement that only earns its build if the corpus proves the static defaults mis-serve the operator. So there is no real "static vs. learned" fork to decide at ship — there is a static mechanism that ships, and a learned layer that is deferred behind evidence. That framing is what makes this affordable.

**On a global endpointing floor bump (explicitly asked): NO — do not bump `endpointing_ms` in Increment 1; keep it at 300.** A global bump is precisely the "always-slow" solution the operator rejected: it taxes every crisp turn. More importantly it *cannot* solve the actual friction, and this is the load-bearing insight of the whole scope. Silence duration alone cannot separate "thinking pause" from "done" — the operator's word-finding pauses and a genuinely-finished crisp sentence produce overlapping trailing-silence distributions. Any static endpointing value large enough to cover his thinking pauses (600-800ms+) would make crisp turns feel sluggish; any value small enough to keep crisp turns snappy (the current 300) fires during his pauses. The discriminator that separates the two is **not silence length, it is the lexical content of the tail** — a trailing "and"/"so"/"because", a filler, a dangling preposition, missing terminal punctuation. That is exactly why adaptivity must live one layer *above* Deepgram, reading the transcript tail, rather than by re-parameterizing a fixed silence timer. Keep the 300ms floor as the snappy commit path; add content-awareness on top.

## 2. Increment 1 — the adaptive mechanism (ships now)

**Architecture:** keep Deepgram `endpointing_ms=300` as the snappy floor (unchanged), and re-interpret every `EVENT_UTTERANCE_END` (both `trigger="speech_final"` and `trigger="utterance_end_fallback"`) as a **candidate** end-of-utterance, not a commit. At the seam (`voice_stt.py:453`), run a pure `classify_tail(text, last_partial)` on the accumulated finals buffer (`" ".join(buffer)`, `voice_stt.py:454`). Complete thought → COMMIT in the same tick (byte-identical to today's fire path). Mid-thought → arm a bounded, concurrent, cancellable HOLD and keep consuming events; fold in any resumed speech; commit only when the hold expires without resumption.

**The hold-and-reopen state machine (pump-local, added alongside `buffer` at `voice_stt.py:443`):**
- `last_partial: str` — updated in the `EVENT_PARTIAL` branch (`voice_stt.py:445-448`).
- `hold_gen: int` — monotonic supersede counter; guards against a resume that races the timer.
- `hold_handle` — the armed `loop.call_later` handle (or a single cancellable `Task`).
- `first_hold_at: float | None` — drives the cumulative ceiling.

Transitions inside the single `async for ev in self._provider.events()` loop:
- **`EVENT_PARTIAL`** → existing `on_partial` behavior, plus `last_partial = ev.text`; if a hold is armed → `_cancel_hold()` (resume detected).
- **`EVENT_FINAL`** → existing `buffer.append(ev.text.strip())` at `voice_stt.py:452` (this is what makes the fold automatic — resumed speech accumulates into the *same* buffer); if a hold is armed → `_cancel_hold()`.
- **`EVENT_UTTERANCE_END`** → the decision point: bypass-and-commit-inline if `self._closing` or `ev.trigger in {"finalize","fake"}` or `not enabled`; sub-`min_utterance_chars` takes the existing `utterance_empty` path (`voice_stt.py:488-493`) with **no hold** (never hold noise); otherwise run `classify_tail` → COMMIT-inline or arm a hold.
- **timer expiry** → `_commit_held(gen)`: `if gen != self._hold_gen: return` (superseded), else commit the accumulated buffer.

**Extract `COMMIT_INLINE`** from today's `voice_stt.py:454-487` (join, clear buffer, PCM snapshot, log, `on_utterance`, shadow) so both the inline path and `_commit_held` call one function — and **move the PCM snapshot from the unconditional `voice_stt.py:459` into `COMMIT_INLINE`** so held audio stays aligned with held text. Both the finals buffer and the teed PCM (`voice_stt.py:379`) keep accumulating until commit, so they cover the full folded utterance and stay in sync — this is required for the shadow corpus to be trustworthy.

**The concurrency constraint is the single sharpest edge and is non-negotiable:** the hold MUST be a concurrent `loop.call_later`/`Task`, never an inline `await asyncio.sleep(...)` at the seam (which would stall consumption of the very partials/finals that signal resumption), and never `asyncio.wait_for(events().__anext__(), timeout)` — the confirmed-this-session bare `await self._events.get()` at `stt_deepgram.py:421` does not catch `CancelledError`, so a timeout-cancel closes the generator and kills STT. Because the loop is single-threaded, the `call_later` callback and the pump body never interleave except at `await` points inside `_commit_held`, which the `hold_gen` check makes race-safe.

**The mid-thought signal set (cheap, last-N-token, no heavy NLP)** — HOLD if any fire on the buffer tail:
- Trailing conjunction: last token ∈ {and, so, but, or, because, if, when, while, then, that, which, as, since, though, although, unless, until, before, after, plus, nor, yet}.
- Trailing filler: last token ∈ {um, umm, uh, uhh, er, hmm, like, well} OR last two tokens ∈ {i mean, you know, kind of, sort of, i guess, let me}.
- Dangling function word: articles/prepositions/determiners/possessives/aux-modals an English sentence essentially never ends on.

COMMIT (the default) when none fire. Terminal punctuation (`text.rstrip()[-1] in {".","?","!"}`, present via `smart_format=true` at `config.py:179`) is a high-confidence completeness cue and vetoes a weak hold. **Reconciled decision on a facet ambiguity: missing terminal punctuation ALONE does NOT trigger a hold** — `smart_format` omits periods on short crisp finals ("yes", "move it to Friday"), so holding on missing-punctuation-alone would tax exactly the crisp turns we must keep snappy. Missing punctuation only *escalates* an already-firing lexical signal. Reuse `barge_in.normalize_text`/`_tokens` (`barge_in.py:59-68`) for the multi-word phrase checks, but use a dedicated single-token strip for the trailing token to avoid `normalize_text`'s apostrophe mangling ("don't"→"don t").

**Bounded extension:** a flat `base_extend_ms` (~500ms) per hold, added on top of the 300ms already elapsed. Holds may stack across successive mid-thought candidate-EOUs (each re-arm requires genuine new speech, so stacking only happens when he *is* still going), but are bounded by a hard cumulative ceiling `max_total_hold_ms` (~1500ms) measured from `first_hold_at`, force-committing regardless of signal when `cum >= max_total_hold_ms`. This gives a deterministic worst-case latency bound and guarantees an "and-um-and-uh" tail can never hang the turn. **Reconciled: keep the ceiling measured from `first_hold_at` across the whole utterance with a generous cap** (Facet A open question; keep-from-first is simpler and the generous ceiling covers legitimate multi-clause thoughts).

**Resume sensitivity — reconciled:** for Increment 1, treat ANY partial/final during a hold as resume (the simple rule both facets lean toward). If Deepgram flush-partials prove empirically noisy, tighten to "partial longer than committed buffer" — noted as a tuning knob, not shipped.

**`is_final`-without-`speech_final` — reconciled:** available today at `stt_deepgram.py:103-104` as a free "paused but not done" signal, but do NOT surface it as a distinct event in Increment 1; accumulated-buffer + `last_partial` are sufficient. Leave it as a documented future weighting input.

**Held/rising intonation — explicitly out of Increment 1.** It is prosodic, not in the transcript text; a PCM-pitch proxy is heavy. The lexical + punctuation + resume signals cover the observed friction (drawn-out speech, word-finding pauses, trailing conjunctions). Accept the flat-lexical-trailing-off-with-rising-pitch case as a known Increment-1 limitation.

**Telemetry ships in Increment 1 (features-only), NO learning applied.** This is the collect-first discipline: emit a durable features-only endpoint event at commit time into the unified corpus. It is passive evidence for the go/no-go; it applies nothing and cannot drift.

**File touchpoints (Increment 1):**
- `src/alfred/web/voice_stt.py` — PRIMARY. `_pump_events` `EVENT_UTTERANCE_END` branch (`:453-493`): `last_partial` tracking, resume-cancel hooks in the PARTIAL/FINAL branches, the `classify_tail` decision + concurrent hold-timer + `hold_gen`, the extracted `COMMIT_INLINE` (with the PCM snapshot moved in from `:459`), `_commit_held(gen)`, and the features-only telemetry emit. Bypass on `_closing`/`{finalize,fake}`; `utterance_empty` keeps no-hold.
- `src/alfred/web/config.py` — new `WebVoiceEndpointHoldConfig` dataclass (`enabled=False`, `base_extend_ms~500`, `max_total_hold_ms~1500`, signal toggles) mirroring `WebSttShadowCaptureConfig` (`:143`); threaded onto `WebVoiceSttConfig` (`:157-181`) as a `field(default_factory=...)`. **Hand-roll construction to dodge the `_build` collision footgun** per CLAUDE.md (a nested block with a key already in `_DATACLASS_MAP` would silently build the wrong dataclass).
- `src/alfred/web/barge_in.py` — reuse `normalize_text`/`_tokens` (`:59-68`) for the pure `classify_tail`; copy `_clamp` (`:197`) + `normalize_barge_settings` (`:201-245`) as the template for a `normalize_endpoint_hold_settings` mount-clamp emitting `web.voice.stt.config_clamped`. No edits to barge itself.

## 3. Increment 2 — per-user learning (deferred behind evidence)

Increment 2 slots endpointing into the **existing V3.1 per-user voice-calibration loop as a second event family — NOT a parallel system** (`docs/v3_1_barge_calibration_scope.md`). This is Facet B's load-bearing decision and I adopt it: **ONE corpus, ONE propose/approve surface.** The justification is physical, not aesthetic — a false-early endpoint fire where the operator talks over Salem's reply onset arrives while Salem is speaking, which is *already* a barge event (`voice_turns.py:551` `_utterance_while_speaking`). The same physical utterance is simultaneously proof the endpointer fired too early *and* a barge candidate; splitting them into two corpora would de-correlate the one signal that matters most.

**Corpus (name it NOW, in Increment 1):** `data/voice_calibration/events.jsonl`, every line carrying `event_family: "barge"|"endpoint"`, `web_user`, `voice_session_id`. This is the one coordination requirement that reaches back into Increment 1: even though Increment 1 applies no learning, its telemetry sink must land under this unified name so it does not fork from V3.1 barge. Whichever of the two (barge Increment 1 or this) ships first creates the corpus; the second generalizes it rather than siblings it.

**Two opposite-sign correction signals** (netted as a signed per-user pressure over ≥3 distinct sessions, never per-event — the exact V3.1 discipline):
- **FALSE-EARLY-FIRE → raise `base_hold_ms`, weight-up the present tail signal.** Detected three ways, joined to the fired EOU by stamping `_last_committed_eou = (utterance_id, clock(), tail_features)` mirroring `_last_suppressed` (`voice_turns.py:147`) and reading it back within the existing 10s window (`voice_turns.py:276`): (a) a new same-user utterance that reads as a grammatical continuation (opens lowercase / with a conjunction / prior tail lacked terminal punctuation); (b) a barge-over-reply-onset — a `web.voice.barge.confirmed`/`late_suppressed` (`voice_turns.py:466/504`) that joins to the just-committed EOU (the cross-family payoff); (c) a short interrupt-phrase barge or `client_cancel` (`voice_turns.py:266`) in the join window.
- **FALSE-HOLD → lower `base_hold_ms`, weight-down the present signal.** A hold armed, expired with zero resumption, and the committed text was already complete — the operator's exact "this user's 'um' is mid-word, not end-of-turn" example maps here (held on a trailing filler that for this user is not an EOU cue → weight 'um' down for this user).

**What is learned:** MVP learns ONE knob — `base_hold_ms` — mirroring V3.1's "learn `too_early_ms` only." Per-user signal **weights** (which trailing-token categories trigger a hold for THIS user) are the deferred second dimension, mirroring V3.1's deferred `echo_threshold` — but the per-signal attribution field is captured from Increment 1 so the data exists when weights are greenlit.

**The loop reuses V3.1 verbatim:** one morning-review Daily Sync calibration tenant listing both families; the same terse reply grammar (`N confirm` / `N reject` / `N 550`); one `_resolve_voice_calibration_correction` resolver with an `event_family` field; one override store `data/voice_calibration.json` keyed by `identity.user` with `from_dict` schema-tolerance and `_meta.from_default` for revert. Two apply sites, one resolver: barge replaces `BargeSettings` at the driver ctor (`routes_voice.py:221`, unchanged); endpoint replaces a new `EndpointHoldSettings` at `_make_stt_worker_factory` (`routes_voice.py:578-628`, worker at `:607`). Both re-clamp on apply, fail-OPEN to the app-global static default on any store parse error (a corrupt calibration file must never brick voice), and log `source=default|calibrated` at session open.

**Per-user key — reconciled as a shared open ruling, not decided here:** `identity.user` (human-readable, `identity.py:57`) vs. `synthetic_chat_id` (non-PII, `identity.py:62`). Recommend `identity.user`, but this is the SAME unresolved V3.1 question and must be decided ONCE for both families before Increment 2 — do not decide it independently for endpointing.

**Go/no-go to greenlight Increment 2 (pre-register now, confirm numbers against real endpoint-event volume after Increment 1 runs):** ≥N confirmed events, consistent-sign net over ≥3 sessions, min effect size, bounded per-cycle delta (~≤150ms per approved cycle — convergence is multi-step, never 400→900 in one jump). If single-user Salem never crosses the bar, **parking after Increment 1 is a legitimate outcome, not a failure** (V3.1 §5) — the dynamic hold alone likely resolves the friction, and the instrument simply stays as passive telemetry.

## 4. The error-bias decision and the latency guarantee

**Two distinct latency semantics, both hard requirements:**

1. **Zero added latency on complete thoughts (structural, always true).** COMMIT is the default decision path; a hold requires a *positive* lexical signal. A crisp turn with no mid-thought token fires at `voice_stt.py:474` in the same tick as today — no timer armed, no scheduling. This is guaranteed four ways, strongest first: default-off; COMMIT-is-default even when enabled; no timer on the snappy path; and bypass rails for `_closing`/`{finalize,fake}`/sub-min-chars noise. This satisfies the operator's hard "do NOT globally slow down" constraint absolutely — the zero-signal path is structurally never held, regardless of anything learned later.

2. **Bounded added latency only in-the-moment (≤ `max_total_hold_ms` ~1500ms).** When a mid-thought signal fires, the hold adds `base_extend_ms` (~500ms) on top of the 300ms already elapsed, capped hard at the cumulative ceiling. Worst case is deterministic and bounded.

**The error-bias — favor false-HOLD over false-FIRE, but only inside the signal-gated exception path.** The observed friction is unambiguously the false-early-fire direction: Salem took the turn during a thinking pause and cut the operator off. That is the costly, relationship-damaging error — it makes Salem feel like it is not listening. A false-hold, by contrast, costs a bounded ≤1500ms of extra silence and nothing else. So when the design is uncertain *within a detected mid-thought moment*, it should err toward holding. **But the bias is asymmetric and gated, not global:** the default (no signal) still favors firing immediately — we do not bias the whole system toward waiting, because that would re-introduce the always-slow tax. The bias applies only *after* a positive lexical signal has already fired: there, accept the occasional false-hold. Concretely this means the ship-time signal lexicon should be **conservative** (err toward under-triggering holds so crisp turns stay snappy), but once a hold is triggered, the extension should be generous enough to actually cover a word-finding pause (hence ~500ms base, not ~200ms). Under-trigger the hold; when you do hold, hold meaningfully. The per-user learner (Increment 2) then tunes both the trigger set and the size against real false-early/false-hold evidence.

## 5. Risks and the honest "a good static heuristic may be enough" possibility

- **The static heuristic may fully resolve the friction — and that is a legitimate terminal state.** This is the reconciled Facet C thesis: the dynamic in-the-moment hold alone likely fixes drawn-out speech + word-finding pauses + trailing conjunctions. On single-user Salem the per-user learner may never cross its go/no-go, and parking after Increment 1 (instrument stays as passive telemetry) is a success, not a shortfall. Do NOT let roadmap-pressure or feature-completeness anxiety fire Increment 2 — only the readout's evidence bar or fresh operator-observed friction does (the deferred-capability friction-trigger rule).
- **Concurrency (sharpest edge).** Confirmed this session: the hold must be a concurrent `call_later`/`Task`; an inline sleep stalls resume detection and `wait_for(events().__anext__(), t)` closes the un-`CancelledError`-catching generator (`stt_deepgram.py:419-424`) and kills STT. The `hold_gen` supersede-check must guard `_commit_held` against a resume that races the timer. This is the one edge that will silently break the stream if done naively — pin it with a test.
- **Snapshot/buffer alignment.** If the PCM snapshot (`voice_stt.py:459`) and buffer-clear are not both moved into `COMMIT_INLINE`, the shadow corpus audio diverges from the held text and the next utterance carries stale audio. Regression-pin the held-then-committed case.
- **False-extend on a genuinely complete utterance** that ends on a filler or lacks smart_format punctuation → unnecessary hold latency. Bounded by the cap; mitigated by the conservative-at-ship lexicon and by not holding on missing-punctuation-alone.
- **Prosody blind spot.** Held/rising intonation is not in the transcript; a flat-lexical trailing-off with rising pitch fires early. Accepted Increment-1 limitation.
- **Barge-direction interaction.** A held final that fires just as Salem begins speaking routes through `submit_utterance`'s speaking gate (`voice_turns.py:535`), which re-checks `_speaking_turn_id` at fire time — partials still flow during the hold so barge Stage-A is unaffected. Pin a regression test that hold-then-fire during an emerging speaking window behaves identically to today.
- **Reconnect/close races.** A hold armed at a reconnect (`_ready.clear()`) or `aclose` must commit-or-drop deterministically; `finalize`/`fake`/`_closing` bypass the hold. Add pins for `trigger=finalize` and the `_closing` bypass.
- **Privacy — the one net-new line.** Live in-process inspection of partial/final tail text is fine (same posture as barge reading `_spoken_text` live). But durable capture must record tail FEATURES ONLY — `trailing_token_is_conjunction`, `trailing_token_is_filler`, `ends_with_terminal_punct`, `n_tokens`, `ms_trailing_silence_at_fire`, `decision`, `hold_ms_applied`, `resumed_within_hold`, and the per-signal attribution category — NEVER the raw tail text. This is one step closer to content than barge's chars/scores-only capture (lexical-CATEGORY booleans), so it is the single net-new privacy consideration and MUST be enforced in the sink schema and independently QA-reviewed before merge. A single accidental raw-tail-text field is a contract violation of the no-transcript-in-logs guarantee (`voice_stt.py:50`, `barge_in.py:11-12`). This QA pass is non-negotiable per the every-ship-gets-independent-QA standard.
- **Corpus-forking risk.** If V3.1 barge Increment 1 ships to `data/barge_corpus/` before this lands, the two calibrations fragment. Mitigation is the co-naming requirement in §3 — name `data/voice_calibration/` NOW, in Increment 1's telemetry sink.
