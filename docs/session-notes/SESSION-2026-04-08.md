### Architectural Reality Check
OpenClaw turned out to be a cloud model router, not a local inference engine. Pivot to direct Ollama integration. But Alfred's existing backends (claude/openclaw/zo) all assume the **agent runtime handles tool use** — none of them are pure-LLM-API backends. Pointing Alfred at Ollama isn't a config change; it requires a new `OllamaBackend` class that:
1. Calls Ollama's `/api/chat` with tool definitions
2. Parses qwen2.5's function calls
3. Executes `alfred vault` commands locally
4. Loops until done
5. Returns BackendResult

Estimated 200-300 lines of new code.

### Decision: Defer Full Integration, Smoke Test Only
Sticking with Claude API for production. Reasons:
- Local LLM swap is infrastructure work, not new capability
- qwen2.5:14b quality is below Claude for nuanced extraction; vault is source of truth
- "Real" local setup will be on the Mac later anyway
- Roadmap items (Morning Brief, multi-instance, Knowledge Alfred) deliver actual user value

## Design Decisions

### Ollama over OpenClaw for Local Models
- OpenClaw is a cloud model router, not a local inference engine — `models list --local` returns empty
- Alfred already has Ollama integration code in the Surveyor embedder (`src/alfred/surveyor/embedder.py`)
- Ollama gives true on-device execution with zero API costs
- OpenClaw remains available as a backend for routing to cloud models if needed

### Model Sizing for 16GB VRAM
- 7B: fits easily, room to spare
- 14B: comfortable with Q4/Q5 quantization (recommended starting point)
- 32B: possible with CPU offload but slower
- Starting with 14B as the sweet spot for quality vs. performance on this hardware
